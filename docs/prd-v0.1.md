# BuildFlow AI PRD v0.1

## 1. 文档信息

- 产品名称：BuildFlow AI
- 版本：v0.1
- 文档类型：MVP PRD
- 阶段目标：只验证一条核心业务链路是否成立
- 产品定位：将模糊产品想法转化为结构化 PRD 的 AI 产品交付助手

## 2. 背景与问题定义

独立开发者、产品经理、学生团队和小型创业团队在产品启动阶段普遍面临以下问题：

- 有想法，但表达不完整，难以快速形成可执行文档
- 需求文档往往从零编写，耗时且质量不稳定
- 大模型能提供灵感，但输出通常冗长、格式飘逸、难以直接进入开发流程
- 项目早期缺少统一输入，导致后续架构、任务拆分、测试和迭代都建立在不稳定基础之上

本阶段不追求“自动完成整个项目”，只验证一个最核心命题：

**用户能否在 10 分钟内，把一个模糊想法沉淀成一份结构化、可继续推进的 PRD。**

## 3. 产品目标

### 3.1 业务目标

- 帮助用户从“产品想法”快速得到“可讨论、可修改、可导出”的结构化 PRD
- 建立后续架构设计、任务拆解、测试生成的稳定输入基础
- 以最小可用范围验证 BuildFlow AI 是否具备长期扩展价值

### 3.2 用户目标

- 用最少输入启动一个项目
- 在 AI 的引导下补齐关键上下文
- 自动获得一份结构清晰的 PRD
- 可以直接导出 Markdown，继续给开发、协作者或自己后续使用

### 3.3 成功标准

- 用户从创建项目到得到第一版 PRD 的总时长小于 10 分钟
- PRD 输出包含固定结构，且字段完整率大于 90%
- 用户无需手工从零写 PRD，最多只需补充关键约束与反馈
- 系统保留完整运行记录，支持后续版本迭代

## 4. 目标用户

### 4.1 核心用户

- 独立开发者
- AI 产品经理候选人 / 初中级产品经理
- 小团队创始人
- 学生项目负责人

### 4.2 用户特征

- 通常已有一个产品想法，但定义不够完整
- 希望借助 AI 提升启动效率，而不是只得到聊天式回答
- 对结果的结构化、可编辑性和后续复用性有明确要求

## 5. MVP 范围定义

### 5.1 本阶段只保留一条主链路

**唯一主链路：Idea Input → Clarification → PRD Generation → Review & Export**

即：

1. 用户输入产品想法和基础背景
2. 系统生成少量澄清问题
3. 用户补充回答
4. 系统生成结构化 PRD
5. 用户在线查看并导出 Markdown

### 5.2 为什么只保留这一条链路

- 它是整个产品交付链路的起点
- 它最能体现“AI 帮助把模糊输入转成结构化输出”的核心价值
- 它能为后续架构、任务、测试生成提供统一输入
- 一天内可落地，且足够形成可演示成果

## 6. 非目标（明确不做）

v0.1 不包含以下能力：

- 不生成技术架构方案
- 不生成 API 设计文档
- 不生成数据库模型
- 不生成任务拆解与工时估算
- 不生成测试用例
- 不做多项目团队协作
- 不做权限系统
- 不做支付系统
- 不做知识库/RAG
- 不做复杂多 Agent 自治编排
- 不做实时协同编辑

这些内容属于后续版本，不进入当前开发范围。

## 7. 核心价值主张

### 7.1 用户价值

- 从零写 PRD 变为基于 AI 引导快速完成
- 从聊天式结果变为结构化产物
- 从一次性灵感输出变为可存档、可复用、可迭代的项目输入

### 7.2 产品价值

- 验证 BuildFlow AI 的最小闭环
- 建立统一的数据模型和文档产物模型
- 为后续扩展版本提供清晰边界

## 8. 核心使用场景

### 场景 A：独立开发者启动新项目

用户有一个粗糙想法，例如“做一个帮助求职者准备 AI 面试的工具”，但不知道如何系统性整理需求。用户希望在较短时间内获得一份像样的 PRD，作为后续开发和展示材料。

### 场景 B：产品经理准备立项草案

用户已经有基本方向，但需要快速整理目标用户、核心功能、成功指标和版本范围，供内部讨论或面试作品展示。

## 9. 用户故事

- 作为一个独立开发者，我希望输入一个产品想法后快速得到澄清问题，以便补齐关键背景。
- 作为一个产品经理，我希望系统帮我生成结构化 PRD，以便我能直接用于讨论和后续推进。
- 作为一个项目发起人，我希望导出 Markdown 文档，以便我能复制到 GitHub、Notion 或文档平台继续使用。

## 10. 单一主链路设计

### 10.1 主链路步骤

#### 步骤 1：创建项目

用户输入：

- 项目名称
- 一句话产品想法
- 目标用户
- 平台类型（Web / Mobile / Both）
- 可选约束（时间、资源、商业目标）

系统行为：

- 创建项目记录
- 保存原始输入
- 跳转到澄清阶段

#### 步骤 2：AI 生成澄清问题

系统根据初始输入生成 3 到 5 个高价值问题，优先补充以下缺失信息：

- 用户痛点
- 核心使用场景
- 最小可行功能范围
- 商业目标或成功标准
- 限制条件

用户行为：

- 逐条回答问题
- 可跳过非关键问题

#### 步骤 3：生成结构化 PRD

系统基于原始输入与澄清回答，生成一份标准化 PRD。输出结构固定，避免自由发挥导致格式漂移。

#### 步骤 4：查看与导出

用户可以：

- 在页面中查看 PRD
- 复制内容
- 导出为 Markdown

## 11. PRD 输出结构

系统生成的 PRD 至少包含以下模块：

1. 产品概述
2. 问题陈述
3. 目标用户
4. 核心场景
5. MVP 目标
6. 功能范围（In Scope）
7. 非目标（Out of Scope）
8. 用户故事
9. 成功指标
10. 风险与待确认项

输出要求：

- 结构固定
- 标题统一
- 内容简洁可读
- 避免空泛术语
- 不生成技术实现细节

## 12. 页面范围

v0.1 只需要 4 个页面/视图：

### 12.1 首页

- 产品介绍
- 创建项目入口

### 12.2 新建项目页

- 输入项目基础信息
- 提交创建项目

### 12.3 澄清问答页

- 展示 AI 生成的问题
- 填写或跳过回答
- 提交进入生成流程

### 12.4 PRD 结果页

- 展示结构化 PRD
- 支持复制与 Markdown 导出

## 13. 功能需求

### 13.1 项目创建

- 支持创建项目
- 支持保存基础字段
- 必填字段校验
- 创建成功后进入澄清阶段

### 13.2 澄清问题生成

- 基于初始输入生成 3 到 5 个问题
- 问题应避免泛泛而谈
- 问题顺序按信息价值排序

### 13.3 回答保存

- 支持提交澄清问题答案
- 支持空值跳过
- 支持与项目绑定保存

### 13.4 PRD 生成

- 生成结构化 PRD
- 结果必须通过 schema 校验
- 结果应与项目、运行记录关联

### 13.5 PRD 展示与导出

- 支持可读性良好的在线渲染
- 支持一键复制
- 支持导出 Markdown 文件内容

## 14. 非功能需求

### 14.1 可维护性

- Prompt、业务编排、数据模型分层管理
- 所有输出结构化，不依赖自由文本解析
- 前后端通过稳定 schema 通信

### 14.2 可测试性

- 项目创建接口可测试
- 澄清问题生成接口可测试
- PRD 生成接口可测试
- PRD schema 可独立测试

### 14.3 可扩展性

- 后续可扩展至 Architecture、Tasks、Tests 模块
- 当前数据模型应允许新增 artifact 类型

### 14.4 性能

- 单次 PRD 生成应在可接受时间内完成
- 首版允许使用轮询获取运行状态

## 17. 验收标准

满足以下条件，即视为 v0.1 PRD 范围定义完成且可进入架构设计：

### 17.1 业务验收

- 用户可以从一个产品想法出发，完成一次从输入到 PRD 导出的闭环
- 输出内容聚焦产品需求，不混入架构、任务和测试模块
- 用户能够清楚理解系统当前解决什么问题，不解决什么问题

### 17.2 文档验收

- PRD 目标单一明确
- 主链路只有一条
- 非目标定义清晰
- 页面范围和功能范围一致
- 数据对象足以支撑下一阶段架构设计

## 18. 风险与约束

- 如果输入信息过少，澄清问题质量可能不稳定
- 如果 Prompt 不做结构化约束，PRD 输出容易漂移
- 如果一次性追求太多模块，项目会重新滑回原型式开发

当前阶段的控制策略：

- 只保留一条主链路
- 只产出一种 artifact：PRD
- 只设计最小数据模型
- 所有输出以固定 schema 为准

## 19. 后续版本规划（仅占位，不进入当前范围）

- v0.2：PRD → Architecture
- v0.3：Architecture → Tasks
- v0.4：Tasks → Test Plan
- v0.5：Feedback-based Revision

## 20. 本阶段结论

BuildFlow AI 的 v0.1 不做“完整产品交付系统”，只做一个最小但完整的验证闭环：

**让用户把一个模糊产品想法，快速转成一份结构化 PRD，并能查看与导出。**

如果这一闭环成立，才继续进入架构设计、任务拆解和测试生成。
